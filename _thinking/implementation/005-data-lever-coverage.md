## 0. 한 줄 요약
데이터 수집 인프라(스크립트·정책)를 다 찾았고, **기존 cap15/20은 완주(expert)를 이미 버리고 충돌만
저장**(취지 일치). 그러나 **우리 정책이 충돌하는 그 지점들이 cap20 데이터에 이미 고속으로 빽빽이 덮여
있음을 실측**(각 ~1만 전이, 14~15 m/s). → "고속 안전 코너 전이 부족" 가설은 **위치 기준 반박됨**.
진짜 갭은 ① **공변량(off-line 상태)** — 기존 데이터 jitter OFF·단일출발·깨끗한 라인뿐, ② **결과 편향**
— 1만 전이가 전부 충돌 ep 안이라 value가 그 라인을 비관. **단순 추가수집 → 효과 의문. jitter ON으로
off-line 커버리지를 노릴 때만 가능성, 그래도 불확실. 표현(H2) 병행 의심.**

## 1. 수집 인프라 (전부 확인)
- **스크립트**: `f1tenth_RL_project/scripts/collect_crash_data.py`
  - stochastic rollout(latent+action 샘플), **충돌 ep만 저장**(완주/diverged/reverse/timeout 폐기).
  - npz 형식 = 우리 로더 입력 그대로(lidar/state/action/reward/log_*/pose/v_max).
  - 옵션: `--start-jitter`(시작 다양화), `--envs N --device cuda --parallel`(배치=빠름), `--max-env-steps`.
- **데이터 생성 정책**(실존):
  - cap15: `runs/cap15_oschersleben/step_105k.pt` @ v_max 15
  - cap20: `runs/stage2_oschersleben/KEEP/KEEP_oscher_lapbest16.6s_step85k_FULL.pt` @ v_max 20
    (= **best lap 16.6s → 2랩 ~33s**, 사용자가 말한 "Dreamer 34초 완주" 정책 본체)
- **★ 완주는 이미 폐기됨**: cap15 500시도 → 충돌 371 저장 / **완주 129 버림**. cap20 → 충돌 291 / **완주 209 버림**.
  생성정책은 26~42% 완주하지만 스크립트가 expert를 버린다 = 사용자 취지(완주 배제)와 설계 일치.

## 2. Q: 기존 데이터에 start-jitter 적용됐나 → **OFF (실측)**
- 로그에 `start_jitter` 출력 없음 + 50 ep 시작 pose **표준편차 [0,0,0]** → 전부 동일 default_pose
  (0.0702, 0.3003, 2.79787) 출발. **시작 다양성 0.**

## 3. Q: "고품질 데이터 확실히 얻나" → 커버리지 실측 (반전)
방법: 센터라인 CSV(`s,x,y`)로 우리 정책 충돌 world좌표→s 매핑, cap15/20 전 transition을 s·속도로
환산해 **하드코너(우리 충돌 s±12m)의 고속(>12 m/s) 안전전이 수**를 셈.

| 우리 충돌 지점(s) | cap20 고속(>12) 안전전이 | 그 지점 cap20 평균속도 |
|---|---|---|
| D2 (s=7m) | **~10,083** | 15.3 m/s |
| D3-300k (s=15m) | **~11,760** | 15.4 m/s |
| D3-280k (s=49m) | **~10,014** | 14.2 m/s |

(cap15도 각 ~9,700~12,000) → **cap20는 그 코너들을 14~15 m/s로 수천 번 무사 통과**. 위치-커버리지 충분.

부가: 기존 충돌 위치 분포(센터라인 lap내%) — cap20는 0-25%(76)·75-100%(125)에 군집, cap15는 25-75% 중앙.
ep 길이 중앙값 cap20=716/cap15=429 (완주2랩 cap20≈1740) → cap20 충돌은 멀리까지 감(고품질 부분궤적은 맞음).

## 4. 진짜 갭 (왜 빽빽한데도 충돌하나)
1. **공변량(covariate) 갭**: 데이터가 **단일 출발 + 깨끗한 라인**(jitter OFF)뿐 → 정책이 미세히 **라인을
   벗어난 off-line 상태**에 빠지면 데이터에 없음 → 복구 불가 → 충돌. *위치는 덮였으나 "벗어난 상태"는 미덮.*
2. **결과 편향(outcome bias)**: 그 1만 전이가 전부 **충돌 ep** 안 → γ=0.999로 충돌(−50) 역전파 → value가
   "그 코너 고속 = 결국 충돌"로 비관. **충돌 데이터를 더 부어도 편향 지속.** 양성-결과 = 완주(expert, 배제).

## 5. 결론 + 다음 옵션
- **단순 추가수집(같은 설정) → 효과 의문** (on-line 커버리지만 증가).
- **데이터 레버를 살리려면 `--start-jitter 0.3~0.5` 필수** — off-line 복구 상태를 덮어 공변량 갭 보강
  (+다양한 far-reaching 충돌이 credit assignment 도움). **단 보장 아님.**
- **표현(H2) 병행 의심**: 빽빽이 덮였는데 못 따라함 = min-pool lidar + feedforward MLP 정밀도/시간맥락 한계
  가능. 후보: frame-stacking(시간맥락), lidar 해상도↑.
- **베팅 결정 필요**: (A) jitter ON 수집 후 재스티칭 vs (B) 표현 업그레이드. (A)가 싸지만 불확실, (B)가
  근본적이나 비쌈(obs 차원 변경→전면 재학습).

## 6. ★ 추가 실측 — lateral(off-line) 커버리지 (jitter 가치 판정)
"위치는 덮였는데 왜 충돌?"의 결정적 근거. 센터라인 법선 기준 **lateral offset**(라인서 벗어난 거리) 측정:

| 우리 충돌 lateral | cap20 데이터 lateral(그 코너 ±5m) | 판정 |
|---|---|---|
| D2(s=7): **−0.76m** | 평균+0.10, std 0.32, max±0.88 | 우리 충돌 = **99%ile 밖** |
| D3-300k(s=15): **−0.69m** | 평균−0.16, std 0.23 | **98%ile 밖** |
| D3-280k(s=49): **+0.73m** | 평균+0.16, std 0.28 | **99%ile 밖** |

→ cap20는 그 코너를 **±0.3m(거의 라인 위)**로만 통과. 우리 정책은 **~0.7m off-line 상태에서 충돌** =
데이터 분포 꼬리(98~99%ile) = **미커버**. **"죽는 그 off-line 상태를 데이터가 안 덮는다"가 실측 확인** =
covariate shift 스모킹건.

### 함의 (정정)
- **start-jitter = #1 후보로 격상.** 위치-커버리지(§3, 1만 전이)는 충분하나 **lateral(off-line) 커버리지가
  좁다**(std 0.3m)는 게 진짜 병목. jitter(±0.5~0.8m 시작 perturbation)가 바로 그 영역을 주입.
- **단서**: jitter는 *시작점*만 흔듦 → 후반 하드코너엔 cap20가 라인 복귀 후 도달할 수 있어 직접 효과 희석.
  보완: (a) 복귀 스킬의 코너 간 transfer 기대, (b) 이상은 중간 perturbation(스크립트 미지원). 그래도 최선의 첫 수.
