# Part I. 문제 정의  `[PDF: 문제 정의]`

## 1. 서론 — 과제·목표
- 1.1 과제: Offline RL — 기존 주행 데이터로 더 빠른 정책 학습. **핵심 제약: 학습 중 환경과 추가 상호작용 없음(순수 offline).**
- 1.2 난관: 폐루프(실주행) 첫 코너 정밀도, 모방학습(BC)이 완주 0%였던 벽.
- 1.3 목표: **G1** 폐루프 2랩 완주 → **G2** 데이터 내 최速 2랩 기록 **cap10 56.14s(결정론 53.66s) 경신.**

## 2. 환경·MDP 정의
- 트랙: Oschersleben(275.18m), 2랩 완주가 과제.
- state(133차원): min-pool 라이다 128 + 차량상태 5(vel_x, vel_y, yaw-rate, prev_steer, prev_speed).
- action(2차원): [조향(steer), 속도(speed)], 정규화 [−1,1] (평가 환경 V_MAX=20).
- reward: 전진(progress, 매 step) + 충돌(−50) + 랩완료(랩당 +100, 2랩 누적 +200).
- 종료조건: 충돌 / 2랩 완주(lap_complete) / 시간초과(timeout).

# Part II. 접근 방법  `[PDF: 접근 방법]`

## 3. 데이터셋 설계 — 왜 충돌 데이터인가
- 3.1 설계 의도: 완주(expert) 시연을 의도적으로 배제하고, **충돌(불완전) 데이터의 조각을 이어붙여(stitching)
  빠른 완주를 합성**한다. (완주 데이터를 넣으면 단순 모방이 되어 "스티칭으로 개선"을 증명할 수 없다.)
- 3.2 구성: cap10_full + cap15 + cap20 + cap15_rc + cap20_rc, **1,687 에피소드**
  (학습 시점 1,023,639 / 최종 보존 1,025,447 transition). 생성 정책 = Dreamer 준-expert(데이터 수집용).
- 3.3 정렬 규약: DreamerV3 정렬(action[t]/reward[t]는 obs[t]를 만든 행동·보상), 로더는 s/a/r/s′/done으로 구성.

## 4. 방법 탐색 — 왜 IQL인가
- 4.1 **LeWM(세계모델) 시도 → 부적합**: 보상 없이 "목표 도달"만 학습하는 구조라 "빠르게"가 목적함수가 아님 +
  입력 불일치(픽셀 vs 라이다) + 계획 시야 0.5초 vs 한 바퀴 6~13초. → 이탈.
- 4.2 **Diffuser(궤적 생성+가치 유도) 시도 → 부적합**: 실주행 평가에서 **재계획 주기 K를 1~30, 유도 강도(scale)를
  0~1까지 두루 바꿔봤으나 전부 충돌·완주 0**(K가 작으면 스핀/후진, 크면 코너 충돌; 모델·정규화·관측은 정상).
  원인 ① 가치가 "빠를수록 좋다(=충돌 직전 고속)"로 치우치고 완주 보상은 할인으로 사라짐, ② 데이터 93%가 충돌이라
  생성 자체가 충돌 경향. **결정적 관찰**: 같은 데이터로 일반 RL 정책은 일부 완주하는데 Diffuser는 0% →
  병목은 데이터가 아니라 **모델**(느린 생성 + 코너 정밀도 한계).
- 4.3 **선택**: 가치 기반 offline RL 중 **TD3+BC와 IQL을 비교 검토 → IQL 선택.**
  - TD3+BC는 critic이 정책의 다음 행동 π(s′)로 부트스트랩해 **데이터에 없는 행동(OOD)도 평가**하고(BC 항으로
    약하게만 억제), IQL은 **데이터 안 행동만으로 가치를 학습(in-sample)해 OOD를 아예 평가하지 않는다.**
  - 우리 데이터는 충돌이 지배적이고 보상이 정규화되지 않아, OOD를 안 보는 IQL이 더 안전·강건.

## 5. IQL 알고리즘
- 5.1 expectile regression(비대칭 손실, τ=0.7)으로 "데이터 안에서 가능한 상위" 가치를 근사 → value/Q 업데이트.
- 5.2 AWR actor 손실 exp(β·advantage), β=3.0 — 이득이 큰 행동을 더 강하게 따라하게.
- 5.3 핵심 성질(양날): OOD를 안 봐서 충돌 데이터에 강건(장점) ↔ 데이터에 없는 주행은 만들어낼 수 없음(한계, 견고성과 연결).

# Part III. 학습 및 실험 방법  `[PDF: 학습 및 실험 방법]`

## 6. 학습 셋업
- 6.1 **순수 offline 보장**: IQL 학습은 환경과 추가 상호작용이 전혀 없다. 데이터셋(센터라인 랜덤 포함)은
  **미리 학습된 behavior 정책으로 수집한 것** = Offline RL의 표준 데이터 구축 단계이며, **학습하는 정책(IQL)은
  학습 동안 환경을 한 번도 돌리지 않는다.** (PDF "추가 상호작용 없이"에 대응.)
- 6.2 데이터 로더: npz → CORL 형식 변환 + action 조화(7.1) 적용.
- 6.3 네트워크·손실 구현: value V / twin Q / Gaussian 정책 MLP(CORL IQL), 손실 = expectile + AWR.
- 6.4 학습 절차: 옵티마이저(Adam)·learning rate·batch, **총 600k step, 체크포인트를 촘촘히 저장(중단돼도 무손실)**,
  학습 환경(전용 venv, torch+cu124, GPU, 백그라운드 실행).
- 6.5 학습 곡선·수렴: 큰 데이터 덕에 과적합을 피했고, 최종(600k) 체크포인트가 best.

## 7. 핵심 기여 (학습을 가능케 한 두 장치)
- 7.1 **action 조화(harmonization)**: cap별 수집 속도상한(10/15/20)을 공통 기준(20)으로 재정규화한다.
  안 하면 같은 정규화 값이 cap마다 다른 실제 속도를 뜻해 **벨만 타깃이 모순**되고 스티칭 자체가 성립하지 않는다.
  → 고속 직선(cap20)과 코너 감속(cap10)을 한 좌표계에서 이어붙일 수 있게 한 전제.
- 7.2 **random-centerline 수집**: 트랙 중앙선의 임의 지점에서 출발하도록 데이터를 모아 **트랙 전역의 진입 상태를
  커버.** 실주행에서 라인을 벗어났을 때 되돌아오는 능력을 학습시켜, 이게 추가되며 처음으로 2랩 완주가 나왔다.
- 7.3 보상·하이퍼: γ=0.999, collision_penalty −50, β=3.0, τ=0.7, common_v_max=20.

## 8. 실험 설계 (ablation) — 무엇이 완주를 여는가
> 데이터·보상 요소를 하나씩 더하며 "속도"와 "완주"가 각각 언제 풀리는지 본다. **핵심: 속도와 완주는 별개 문제다.**
- (1) 느린 데이터(cap10)만 → 9.5 m/s로 느림, **lap1은 돌지만 lap2에서 충돌 = 2랩 미완주.**
- (2) + 빠른 충돌 데이터(cap15/20) → 16 m/s·한 바퀴 18s로 **속도는 풀림**, 그러나 **여전히 lap2 충돌 = 2랩 미완주.**
- (3) + 충돌 페널티 강화(−100) → 충돌 위치만 옮겨갈 뿐 완주 0 = **보상 튜닝으로는 못 푼다.**
- (4) + 센터라인 랜덤 수집 → **처음으로 2랩 완주(34.46s).**
- 결론: 속도 문제는 (2)가, 완주(견고성) 문제는 (4)의 random-centerline이 풀었다. cap 데이터만으로는 둘 다 lap2를 못 넘었다.

## 9. 평가 프로토콜
- 평가 환경 V_MAX=20, 결정론 정책, 완주 판정 = 2랩 lap_complete, 2랩 lap time 측정.
- seed = 라이다 스캔 노이즈(출발 위치는 동일). **best 선정 = 30개 체크포인트 스윕 + seed 스윕.**
- 재현성: **seed 고정 시 100% 재현**, 단 평가 실행 간 lap time은 34.32~34.46s로 미세 변동.

# Part IV. 성능 및 결과 분석  `[PDF: 성능 및 결과 분석]`

## 10. 결과
- 10.1 최종 성능: **2랩 34.46s, lap [17.98, 16.48]**(궤적 플롯 포함). 데이터 최速 cap10 56.14s 대비 **+21.68s**,
  결정론 기록 53.66s도 경신. 단일 best lap 16.48s ≈ Dreamer expert 16.6s(동급), 2랩 ~33s에는 **근접(능가 아님).**
- 10.2 견고성(정직하게, 당당히): 완주 **체크포인트 4/30**(300k·360k·420k·600k), seed **3/4**(0·2·3 완주, 1 충돌),
  취약점 = 센서 노이즈. **희소하지만 학습 전반(300k~600k)에서 재현돼 우연이 아니며**(cf. DreamerV3가 매우 낮은
  확률로 다이아몬드 채굴에 성공한 것도 성공으로 인정됨), **추가 견고성은 본 목표 달성에 불필요하다 판단해 여기서 종료.**

## 11. 논의
- 11.1 데이터의 정직한 양면성: 충돌 라벨의 **99.84%가 정상 주행** 전이, 충돌 ep의 27%가 1랩 완주 후 충돌,
  랩완료 보상 511건 내포, 평균 12.9 m/s. ↔ 반대편: 모든 전이가 충돌로 끝나고, **데이터엔 고속 2랩 완주 시연이 0개**
  (이게 스티칭 성과를 돋보이게 하는 지점).
- 11.2 "데이터 매칭이 아니라 능가": 능가 대상 = cap10 데이터 기록(56.14s)과 임의의 단일 궤적. 저커버리지 구간에선
  스티칭이 붕괴할 수 있다는 우려가 있으나 **34.46s 실측이 이를 반증.**
- 11.3 실패에서 얻은 일반화 교훈: 보상 튜닝 무효(문제는 보상 *크기*가 아니라 *거리/할인*), covariate(상태 커버리지)가
  진짜 병목, 생성형(planner)과 정책형(policy)의 코너 정밀도 차이.
- 11.4 한계: (1) **경계선 견고성**(§10.2) — 데이터 확장 등으로 개선 여지가 있어 보이나 미검증. (2) **외삽 한계** —
  in-sample offline RL은 데이터 support 밖 주행을 만들어내지 못하므로, 목표 영역(고속에서의 안전 완주) 상태
  커버리지가 얇은 것이 견고성 상한과 연결된다(우리 데이터가 전체 상태공간을 대변하지 않음).

## 12. 향후 계획
- 12.1 완주 토대(cap10) 없이 **전부 1랩 미만 데이터만으로도** 완주를 합성할 수 있는가.
- 12.2 여러 맵의 주행 데이터로 학습하면 일반화 성능이 오르는가(현재는 단일맵 Oschersleben).
